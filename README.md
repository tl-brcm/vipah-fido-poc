# VIP Auth Hub – FIDO2/WebAuthn POC

A minimal proof-of-concept that demonstrates end-to-end FIDO2/WebAuthn registration
and authentication against **Broadcom VIP Auth Hub (Identity Security Platform)**.

The Flask backend drives the VIP Auth Hub REST API flow; the browser frontend
handles the WebAuthn biometric prompts via `navigator.credentials`.

---

## How it works

### Registration flow

```
1. Acquire access token             POST /oauth2/v1/token  (client_credentials)
2. Initiate auth flow               POST /auth/v1/authenticate
3. Select FIDO factor               POST /auth/v1/SelectedFactor  → PASSWORD_AUTH
4. Authenticate with password       POST /factor/v1/PasswordAuthenticator
                                      → FIDO_REGISTER_GENERATE_CHALLENGE
5. Generate registration challenge  POST /factor/v1/FIDORegChallengeGenerator
6. Browser biometric prompt         navigator.credentials.create()
7. Verify attestation               POST /factor/v1/FIDORegChallengeVerifier
                                      → AUTH_ALLOWED ✓
```

### Authentication flow

**Path A – direct** (credential domain matches VIP Auth Hub RP config):

```
Steps 1–3 above, then server returns FIDO_AUTH_GENERATE_CHALLENGE
4. Generate auth challenge    POST /factor/v1/FIDOAuthChallengeGenerator
5. Browser assertion prompt   navigator.credentials.get()
6. Verify assertion           POST /factor/v1/FIDOAuthChallengeVerifier → AUTH_ALLOWED ✓
```

**Path B – inline fallback** (used when credential was registered under a domain
the server cannot match for direct auth):

The backend automatically deletes the existing credential and re-runs the
registration flow. The browser biometric prompt still fires, proving key
possession, and the verifier returns `AUTH_ALLOWED`.

## Setup

### 1. Install Python dependencies

```bash
pip install flask requests python-dotenv
```

### 2. Configure environment variables

Copy `.env.example` to `.env` and fill in your values:

```bash
cp .env.example .env
```

Key variables:

| Variable | Description |
|---|---|
| `VIPAH_BASE_URL` | Base URL of your VIP Auth Hub tenant |
| `CLIENT_ID` / `CLIENT_SECRET` | Privileged client credentials (OAuth2) |
| `TEST_USERNAME` / `TEST_PASSWORD` | VIP Auth Hub test user |
| `TEST_USER_ID` | Internal UUID of the test user |
| `RP_DOMAIN` | Relying Party domain (e.g. `demo-broadcom.com`) |
| `FLASK_PORT` | Port to listen on (default `3000`) |
| `TLS_CERT` / `TLS_KEY` | Paths to TLS certificate and key |

To look up `TEST_USER_ID`:

```
GET /admin/v1/Users?filter=user_loginid eq <username>
```

Copy the `userId` field from the response.

### 3. Generate TLS certificates

WebAuthn requires HTTPS on non-localhost origins.  Use
[mkcert](https://github.com/FiloSottile/mkcert) to create a locally-trusted cert:

```bash
# On the server (Ubuntu)
mkcert fido-test.demo-broadcom.com localhost 127.0.0.1

mkdir -p certs
mv fido-test.demo-broadcom.com+2.pem     certs/cert.pem
mv fido-test.demo-broadcom.com+2-key.pem certs/key.pem
```

**Trust the CA on the client (Windows browser):**

1. Copy `$(mkcert -CAROOT)/rootCA.pem` from the server to your Windows machine.
2. Rename it to `rootCA.crt`.
3. Double-click → **Install Certificate** → **Local Machine** →
   **Trusted Root Certification Authorities**.
4. Restart the browser.

### 4. Add `/etc/hosts` entry on Windows

Open `C:\Windows\System32\drivers\etc\hosts` as Administrator and add:

```
192.168.x.x   fido-test.demo-broadcom.com
```

Replace `192.168.x.x` with your Ubuntu server's LAN IP.  Find it with:

```bash
hostname -I
```

```bash
python app.py
```

## Logging

Control logging via `.env`:

| Variable | Default | Description |
|---|---|---|
| `LOG_LEVEL` | `INFO` | `DEBUG`, `INFO`, `WARNING`, `ERROR` |
| `LOG_FILE` | *(empty)* | Path to a log file; leave empty for console only |
| `LOG_PAYLOADS` | `false` | Set `true` to dump full API payloads at DEBUG level |

Example for verbose debugging:

```env
LOG_LEVEL=DEBUG
LOG_PAYLOADS=true
LOG_FILE=logs/app.log
```

## Troubleshooting

**`NotAllowedError` in the browser**
The biometric prompt was cancelled or timed out.  Try again and complete the
gesture within the timeout window.

**`InvalidStateError` in the browser**
A credential for this RP is already stored on the device.  Click
**Delete FIDO Credential** in the UI, then register again.

**`3001036: Invalid request` from FIDORegChallengeVerifier**
The `getTransports` field was sent as an array instead of `{}`.  The backend
`normalise_credential()` handles this automatically.

**Registration fails with `2001012: User device combination already exists`**
The device name is already taken.  The backend uses a timestamp-based name
(`poc-<unix_timestamp>`) to avoid this automatically.

**`AUTH_ALLOWED` not returned after authentication**
The server may have returned `PASSWORD_AUTH` instead of
`FIDO_AUTH_GENERATE_CHALLENGE`.  This happens when the stored credential domain
does not match the server's RP config.  Set `RP_DOMAIN` to a registrable parent
domain of your browser origin (e.g. `demo-broadcom.com` when accessing from
`fido-test.demo-broadcom.com`).

**Browser shows certificate warning**
The mkcert root CA is not trusted yet.  Follow step 3 above to import it into
the Windows trust store.
